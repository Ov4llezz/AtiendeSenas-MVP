****Infraestructura de Entrenamiento e Integración Híbrida***
Para el entrenamiento del modelo, se implementó una arquitectura de procesamiento híbrida que desacopla la interfaz de desarrollo del motor de cómputo. Se utilizó Google Colab como interfaz de usuario (Frontend/IDE) debido a sus capacidades de visualización y documentación interactiva, mientras que la carga computacional fue ejecutada en una instancia de máquina virtual (VM) dedicada en Google Cloud Platform (GCP).

La integración entre el entorno local (Colab) y el servidor remoto (GCP g2-standard-8 con GPU NVIDIA L4) se estableció mediante un protocolo de conexión segura y compatibilidad de backends, detallado a continuación:

X.X.1 Túnel SSH y Reenvío de Puertos (Port Forwarding)
La comunicación entre el cliente local y la instancia remota se aseguró mediante un túnel SSH encriptado. Se utilizó la técnica de reenvío de puertos local (local port forwarding), mapeando un puerto arbitrario de la máquina local (9999) hacia el puerto de servicio de Jupyter en la VM (8888).

El comando de infraestructura utilizado fue: gcloud compute ssh ... -- -L 9999:localhost:8888

Esto permite que las peticiones HTTP enviadas a localhost:9999 sean encapsuladas a través del protocolo SSH y entregadas de manera segura al servidor de inferencia en la nube, mitigando la necesidad de exponer puertos críticos a la red pública de internet.

X.X.2 Gestión de Compatibilidad del Entorno Jupyter
Para habilitar la conexión con el entorno de ejecución de Google Colab ("Local Runtime"), fue necesario realizar una degradación controlada (downgrade) de la versión del servidor Jupyter y la instalación de extensiones de transporte específicas:

Estandarización de Versión (notebook<7.0.0): Se restringió la versión del paquete notebook a la serie 6.x. Esto se debe a que la arquitectura actual de Google Colab utiliza protocolos de comunicación legacy que no son nativamente compatibles con la arquitectura asíncrona introducida en Jupyter Notebook 7.0+.

Protocolo de Transporte (jupyter_http_over_ws): Se implementó la extensión Jupyter HTTP over WebSocket. Dado que los navegadores modernos restringen las conexiones directas entre orígenes web seguros (HTTPS) y servidores locales no seguros, esta extensión actúa como una capa de interoperabilidad, permitiendo el flujo de datos bidireccional mediante WebSockets autenticados entre la interfaz web de Colab y el kernel remoto.

X.X.3 Políticas de Seguridad de Origen Cruzado (CORS)
Finalmente, para autorizar la inyección de código y la recepción de salidas desde el dominio de Google, se configuró el servidor Jupyter con parámetros explícitos de Cross-Origin Resource Sharing (CORS):

Parámetro: --NotebookApp.allow_origin='https://colab.research.google.com'

Justificación: Esta directiva modifica la política de seguridad del servidor para aceptar peticiones provenientes exclusivamente del dominio de investigación de Google, rechazando cualquier otro intento de conexión externa no autorizada, garantizando así la integridad del entorno de entrenamiento.
